# Digital Forensics Methodology & Evidence Handling

**Version:** 1.0
**Date:** June 2026
**Role:** Digital Forensics Analyst / Incident Responder

---

## 1. Overview

Digital Forensics is the practice of collecting, preserving, analyzing, and presenting digital evidence in a manner that is legally admissible and scientifically defensible. In the context of our SOC, forensics is invoked during or after an incident to answer: **What happened, when, how, and by whom?**

This document establishes the SOC's standard procedures for evidence handling, following the principles outlined in NIST SP 800-86 (Guide to Integrating Forensic Techniques into Incident Response).

---

## 2. The Four Phases of Digital Forensics

### Phase 1: Collection
Gather all relevant digital evidence without altering it.

**Key Principles:**
- **Order of Volatility:** Collect the most volatile evidence first. RAM disappears when the machine powers off; disk artifacts persist.
  1. CPU Registers / Cache (most volatile)
  2. RAM (Running processes, network connections)
  3. Swap / Page Files
  4. Disk (File system, logs, registry)
  5. Removable Media
  6. Network Logs (SIEM copies, firewall logs)
  7. Backup Tapes (least volatile)

- **Never work on the original evidence.** Always create a forensic image (bit-for-bit copy) and work on the copy.

**Tools for Collection:**
| Tool | Purpose |
| :--- | :--- |
| `FTK Imager` (free) | Create forensic disk images (E01 format) |
| `KAPE` (free) | Rapid triage collection of key artifacts (registry, event logs, prefetch) |
| `dd` (Linux) | Bit-for-bit disk imaging on Linux systems |
| `Volatility` (free) | RAM analysis (process list, network connections, injected code) |

### Phase 2: Preservation
Ensure evidence integrity so it can be used in legal proceedings.

**Hash Verification:**
After collecting an evidence file, compute its cryptographic hash. This hash acts as a digital fingerprint—if the evidence is altered in any way, the hash will change.

```bash
# Compute SHA-256 hash of an evidence file
sha256sum evidence_disk_image.dd

# Example output:
# a1b2c3d4e5f6...  evidence_disk_image.dd
```

**Chain of Custody:**
Every time evidence changes hands (e.g., from the SOC Analyst to the Legal team), it must be documented.

| Field | Description |
| :--- | :--- |
| **Evidence ID** | Unique identifier (e.g., `EVD-2026-001`) |
| **Description** | "Forensic image of WKS-WIN-01 C: drive" |
| **Collected By** | Name, title, date, time |
| **Hash (SHA-256)** | `a1b2c3d4e5f6...` |
| **Storage Location** | "Forensics Lab, Locker B, Shelf 2" |
| **Transferred To** | Name, title, date, time, reason |

### Phase 3: Analysis
Examine the preserved evidence to reconstruct the attack timeline.

**Key Artifacts on Windows:**
| Artifact | Location | What It Reveals |
| :--- | :--- | :--- |
| **Event Logs (EVTX)** | `C:\Windows\System32\winevt\Logs\` | Logins, process execution, service installs |
| **Prefetch Files** | `C:\Windows\Prefetch\` | Programs that were executed (even if deleted) |
| **Registry Hives** | `C:\Windows\System32\config\` | Persistence keys, USB history, user activity |
| **$MFT** | Root of NTFS volume | Master File Table—every file ever created, even deleted ones |
| **Amcache** | `C:\Windows\AppCompat\Programs\Amcache.hve` | SHA-1 hashes of executed programs |
| **ShimCache** | Registry `SYSTEM` hive | Evidence of program execution |

**Key Artifacts on Linux:**
| Artifact | Location | What It Reveals |
| :--- | :--- | :--- |
| **Auth Logs** | `/var/log/auth.log` | SSH logins, sudo usage |
| **Bash History** | `~/.bash_history` | Commands typed by the attacker |
| **Crontab** | `/etc/crontab`, `/var/spool/cron/` | Persistence via scheduled tasks |
| **Last Log** | Output of `lastlog` / `last` commands | Last login timestamps per user |
| **Web Server Logs** | `/var/log/nginx/access.log` | Requests to web shells, exploit payloads |

### Phase 4: Reporting
Present findings in a clear, objective, factual report.

**Report Structure:**
1. Executive Summary (non-technical)
2. Scope of Investigation
3. Evidence Collected (with hashes)
4. Timeline of Events (chronological)
5. Indicators of Compromise (IOCs)
6. Findings and Conclusions
7. Recommendations
8. Appendices (raw logs, screenshots)

## 3. IOC (Indicator of Compromise) Documentation

After completing an investigation, all discovered IOCs must be documented and shared with the SOC for integration into detection rules and blocklists.

| IOC Type | Example | Action |
| :--- | :--- | :--- |
| **IP Address** | `185.143.223.47` | Block at firewall; add to threat intel feed |
| **Domain** | `evil-c2-server.xyz` | Block at DNS firewall |
| **File Hash (SHA-256)** | `e3b0c44298fc1c149...` | Add to EDR/AV blocklist |
| **Email Address** | `phishing@attacker.com` | Block at email gateway |
| **Registry Key** | `HKLM\...\Run\malware` | Add to detection rule (Sysmon EID 13) |
