**Version:** 1.0

**Role:** Red Team / Purple Team Operator

## 1. Overview
Writing detection rules is only half the battle. A mature SOC must perform **Purple Teaming**—the process of continuously testing defenses against simulated attacks to ensure rules trigger successfully. 

This document provides the standard operating procedures for emulating adversaries using Kali Linux (for network-based attacks) and Atomic Red Team (for host-based execution).

---

## 2. Emulation Environment Setup

### 2.1 Atomic Red Team (Windows Execution)
Atomic Red Team (ART) is a library of simple, open-source tests mapped to the MITRE ATT&CK framework. It allows us to safely simulate malware behaviors on our Windows target (`WKS-WIN-01`).

**Installation (Run as Admin in PowerShell on `WKS-WIN-01`):**
```powershell
# Set execution policy
Set-ExecutionPolicy Bypass -Scope CurrentUser -Force

# Install the Invoke-AtomicRedTeam module
IEX (IWR 'https://raw.githubusercontent.com/redcanaryco/invoke-atomicredteam/master/install-atomicredteam.ps1' -UseBasicParsing);
Install-AtomicRedTeam -getAtomics
```

### 2.2 Kali Linux (Network Attacks)
Ensure the Kali Linux VM (`SIM-KALI-01`) is on the same subnet as the target machines (`192.168.100.0/24`) and has standard tools (Hydra, Nmap, Metasploit) installed.

## 3. Emulation Scenarios

The following scenarios are designed to explicitly trigger the detection rules created in Phase 5.

### Scenario 1: Credential Access (Brute Forcing)
**Objective:** Trigger Detection Rule 1.02 (SSH Brute Force).
**Attacker Machine:** Kali Linux (`SIM-KALI-01`)
**Target Machine:** Ubuntu Server (`SRV-LNX-01` : `192.168.100.30`)

**Execution Command (Kali):**
```bash
# Create a dummy password list
echo -e "password\nadmin123\nroot\n123456\nletmein" > passwords.txt

# Run Hydra against the target
hydra -l root -P passwords.txt ssh://192.168.100.30
```
**Expected SOC Result:** Filebeat ships `/var/log/auth.log`. Kibana detects $>20$ failed SSH attempts from `SIM-KALI-01`.

---

### Scenario 2: Defense Evasion (Event Log Clearing)
**Objective:** Trigger Detection Rule 4.01 (Event Log Clearing).
**Attacker Machine:** Windows 10 (`WKS-WIN-01`)
**Tool:** PowerShell / Wevtutil

**Execution Command (PowerShell):**
```powershell
# Clear the Application event log (requires Admin)
wevtutil cl Application

# Clear the Security event log (requires Admin)
wevtutil cl Security
```
**Expected SOC Result:** Winlogbeat captures Event ID 1102. High severity alert fires in Kibana.

---

### Scenario 3: Execution (Obfuscated PowerShell)
**Objective:** Trigger Detection Rule 2.01 (Suspicious Encoded PowerShell).
**Attacker Machine:** Windows 10 (`WKS-WIN-01`)
**Tool:** Atomic Red Team (T1059.001)

**Execution Command (PowerShell):**
```powershell
# Execute the specific Atomic Red Team test for Base64 execution
Invoke-AtomicTest T1059.001 -TestNumbers 1
```
*(Alternatively, run manually):*
```powershell
powershell.exe -enc ZWNobyAiaGVsbG8gd29ybGQi
```
**Expected SOC Result:** Sysmon captures Event ID 1. The `process.command_line` contains `-enc`. High severity alert fires.

---

### Scenario 4: Persistence (Scheduled Tasks)
**Objective:** Trigger Detection Rule 3.01 (Scheduled Task Creation).
**Attacker Machine:** Windows 10 (`WKS-WIN-01`)
**Tool:** CMD

**Execution Command (CMD):**
```cmd
schtasks /create /tn "MaliciousUpdateTask" /tr "C:\Users\Public\malware.exe" /sc daily /st 12:00
```
**Expected SOC Result:** Sysmon EID 1 captures `schtasks.exe /create`. Alert fires.

---

### Scenario 5: Defense Evasion (Disabling Defender)
**Objective:** Trigger Detection Rule 4.02 (Windows Defender Disabled).
**Attacker Machine:** Windows 10 (`WKS-WIN-01`)
**Tool:** PowerShell

**Execution Command (PowerShell):**
```powershell
Set-MpPreference -DisableRealtimeMonitoring $true
```
**Expected SOC Result:** Sysmon EID 1 captures the specific PowerShell cmdlet. Alert fires.

---

## 4. SOC Validation & Eradication
After executing these tests, the SOC Analyst must:
1. Log into Kibana.
2. Verify that the expected alerts triggered.
3. If an alert **did not** trigger, investigate why (e.g., was Sysmon not running? Was the KQL query typoed?).
4. Execute the eradication steps (e.g., delete the scheduled task, re-enable Windows Defender).
