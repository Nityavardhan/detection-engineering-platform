# Project Documentation Standards & Style Guide

**Version:** 1.0
**Date:** June 2026
**Role:** Technical Writer / SOC Engineer

---

## 1. Overview

All documents within this SOC Portfolio follow enterprise consulting standards. This guide ensures consistency across every phase, making the portfolio recruiter-ready and interview-presentable.

---

## 2. Document Structure Template

Every major document follows this structure:

```
# Document Title

**Version:** X.X
**Date:** Month Year
**Role:** Primary audience / author role
**Classification:** Internal / Confidential

---

## 1. Overview / Executive Summary
Brief, non-technical summary of the document's purpose.

## 2. Scope
What this document covers and does not cover.

## 3–N. Core Content Sections
Detailed technical content, organized logically.

## N+1. Recommendations
Actionable next steps.

## Appendix (if needed)
Supporting data, raw logs, screenshots.
```

---

## 3. Formatting Standards

### 3.1 Headings
- `#` — Document title (used once per document)
- `##` — Major sections (numbered: 1, 2, 3…)
- `###` — Subsections (numbered: 1.1, 1.2, 2.1…)

### 3.2 Tables
- Use Markdown tables for structured data (asset inventories, risk registers, mappings).
- Always include a header row.
- Align columns with `:---` (left), `:---:` (center), or `---:` (right).

### 3.3 Code Blocks
- Use fenced code blocks with language identifiers (```kql, ```bash, ```powershell).
- Always explain what a command does *before* showing the command.

### 3.4 Terminology
- Use formal, enterprise language. Say "endpoint" not "computer." Say "exfiltration" not "data theft."
- Abbreviations must be defined on first use (e.g., "Mean Time to Detect (MTTD)").

### 3.5 Cross-References
- When referencing another phase document, use the format: *"See Phase 5, Rule 2.01 for details."*
- When referencing a policy, use: *"As mandated by the Logging and Monitoring Policy (Phase 10)."*

---

## 4. File Naming Convention

```
[Phase Number].[Document Number]_[Descriptive_Name].md
```

Examples:
- `1.0_Project_Charter.md`
- `5.0_Detection_Library_Part1.md`
- `10.2_Security_Policies.md`

---

## 5. Version Control

All documents are version-controlled via Git. Commit messages follow:

```
Phase [N]: [Brief description of change]
```

Examples:
- `Phase 5: Add detection rules for credential access`
- `Phase 10: Create 7 enterprise security policies`

---

## 6. Quality Checklist

Before publishing any document, verify:

- [ ] Document has Version, Date, and Role metadata.
- [ ] All technical terms are explained on first use.
- [ ] All detection rules reference MITRE ATT&CK IDs.
- [ ] All policies reference the compliance framework they satisfy.
- [ ] All code blocks have language identifiers.
- [ ] No placeholder text (e.g., "[TODO]", "TBD") remains.
- [ ] Document has been reviewed for spelling and grammar.
