# Log Collection Strategy: Windows Environment

**Version:** 1.0
**Date:** June 2026
**Role:** SOC Analyst / Detection Engineer

---

## 1. Overview
In a SOC, logging everything is too expensive and inefficient. We must adhere to a strict **Log Collection Strategy** that prioritizes high-fidelity events capable of identifying malicious behaviors (TTPs). This document outlines the critical Windows Event IDs (EIDs) we ingest via Winlogbeat and Sysmon, and *why* they matter.

---

## 2. Windows Native Security Logs

These logs are generated by the Windows OS and provide visibility into user authentication, privilege escalation, and object access.

### 2.1 Authentication & Logon Activity
Attackers frequently steal credentials (e.g., via Phishing or Mimikatz) and attempt to log in laterally (Pass-the-Hash, RDP).

| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **4624** | Successful Logon | Tracks who logged in, from what IP, and how (Logon Type). Logon Type 3 (Network) and 10 (RDP) are heavily monitored for lateral movement. |
| **4625** | Failed Logon | High volumes of 4625 from a single source indicate Brute Force or Password Spraying attacks (MITRE T1110). |
| **4648** | Logon with explicit credentials | Often generated when an attacker uses `runas.exe` to execute commands as an Administrator. |
| **4769** | Kerberos Ticket Requested | Abnormal ticket requests can indicate Kerberoasting or Golden Ticket attacks. |

### 2.2 Account Management
Attackers often create local accounts or add themselves to the Administrators group to maintain persistence.

| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **4720** | User Account Created | Any unexpected account creation is a massive red flag for persistence (MITRE T1136). |
| **4732** | Member Added to Security-Enabled Local Group | Specifically monitored when a user is added to the `Administrators` or `Remote Desktop Users` groups. |

### 2.3 Defense Evasion
When adversaries land on a machine, they try to hide their tracks.

| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **1102** | Audit Log Cleared | If the Security log is cleared, it is almost certainly a human attacker trying to cover their tracks (MITRE T1070.001). |
| **7045** | New Service Installed | Attackers frequently install malicious services (e.g., PsExec) to execute payloads as `SYSTEM`. |

## 3. Sysmon Logs (Advanced Telemetry)

While Native Windows logs are great for authentication, they lack detail on exactly *what* a process did. Sysmon provides deep execution, network, and registry telemetry.

### 3.1 Process Creation (Execution)
| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **Sysmon 1** | Process Creation | Logs every process that starts, including the exact command-line arguments and the parent process that spawned it. Crucial for detecting malicious PowerShell or Living-off-the-Land (LOLBins) binaries. |
| **Sysmon 10** | Process Access | Tracks when one process opens a handle to another. Heavily used to detect credential dumping (e.g., Mimikatz accessing `lsass.exe`). |

### 3.2 Network & File Integrity
| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **Sysmon 3** | Network Connection | Associates a network connection directly with a process (e.g., if `powershell.exe` connects to an external Russian IP, it's highly suspicious). |
| **Sysmon 11** | File Create | Monitors when a process drops a new executable file onto the disk (e.g., dropping malware into the `C:\Users\Public\` folder). |

### 3.3 Persistence Mechanisms
| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **Sysmon 12/13/14** | Registry Events | Detects when an attacker modifies the registry, particularly targeting "Run" keys to ensure their malware starts automatically when the computer reboots (MITRE T1547.001). |

---

## 4. PowerShell Logs

PowerShell is an attacker's favorite tool because it is built-in and incredibly powerful. We must enable **Script Block Logging** via Group Policy to see exactly what scripts are running.

| Event ID | Name | Why It Matters for SOC |
| :--- | :--- | :--- |
| **4104** | Script Block Logging | Captures the full content of a PowerShell script as it executes. This allows us to detect obfuscated or encoded payloads (e.g., `powershell -enc <base64>`). |

## 5. Conclusion
By selectively forwarding these specific Event IDs, our SIEM remains highly performant, our storage costs remain low, and our SOC analysts have exactly the high-fidelity context they need to investigate incidents.
