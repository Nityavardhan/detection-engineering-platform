# Threat Hunting Playbook Library

**Version:** 1.0
**Date:** June 2026
**Role:** Threat Hunter / SOC Analyst (Tier 2+)

---

## 1. What is Threat Hunting?

Detection rules are **reactive**—they wait for something bad to happen and then alert you. Threat Hunting is **proactive**—you assume the adversary has already bypassed your rules and you go looking for them.

Every hunt follows this cycle:

1. **Hypothesis** — Start with a theory (e.g., "An attacker may be using encoded PowerShell to evade our rules").
2. **Data Collection** — Identify which log sources contain the evidence.
3. **Query & Investigation** — Write KQL queries to search for anomalies.
4. **Findings** — Document what you discovered.
5. **Conclusion & Recommendations** — Did you find evidence? If yes, escalate. If no, document the hunt as a negative finding (which is still valuable).

## Hunt 1: Abnormal PowerShell Execution by Non-Technical Users

**Hypothesis:** Adversaries may have compromised a non-technical user's account (e.g., HR, Finance) and are using PowerShell for reconnaissance or data staging. Standard corporate users should almost never run PowerShell.

**Data Source:** Sysmon Event ID 1 (Process Creation)

**KQL Query:**
```kql
event.code: 1 AND process.name: "powershell.exe" AND NOT user.name: ("admin" OR "svc_*" OR "SYSTEM")
```

**Expected Findings:** A list of PowerShell executions by standard user accounts. Review the `process.command_line` for each to determine intent.

**Investigation Steps:**
1. Review the command-line arguments. Is it a standard IT script or something suspicious?
2. Check the parent process (`process.parent.name`). If PowerShell was spawned by `outlook.exe` or `winword.exe`, it is highly suspicious (possible macro execution).
3. Cross-reference the user account with HR to confirm the individual's role.

**Conclusion Template:** "Hunt completed on [DATE]. Reviewed [N] PowerShell executions by non-admin users. [N] were benign IT automation. [N] require further investigation. Escalated [ticket IDs]."

## Hunt 2: Beaconing Activity (Command & Control)

**Hypothesis:** An implant on a compromised host may be "beaconing" (making periodic outbound connections at regular intervals) to a Command & Control (C2) server.

**Data Source:** Sysmon Event ID 3 (Network Connection), Firewall Logs

**KQL Query:**
```kql
event.code: 3 AND NOT destination.ip: ("10.*" OR "192.168.*" OR "172.16.*") | stats count() by process.name, destination.ip | where count > 50
```

**Expected Findings:** Processes making a suspiciously high number of connections to a single external IP.

**Investigation Steps:**
1. Identify the `destination.ip` and query it against Threat Intelligence (VirusTotal, AbuseIPDB).
2. Check the `process.name`. If it is `svchost.exe` or `rundll32.exe` contacting an external IP, treat it as a critical finding.
3. Analyze the timing interval between connections. C2 beacons often fire every 30–60 seconds with minor jitter.

**Conclusion Template:** "Hunt completed on [DATE]. Identified [N] external connections by suspicious processes. [N] IPs flagged as malicious by Threat Intelligence. Escalated to Incident Response."

## Hunt 3: Lateral Movement via RDP from Unusual Sources

**Hypothesis:** An adversary who has compromised one host may be using RDP to pivot to other machines in the network. Legitimate RDP should only originate from the IT administration subnet.

**Data Source:** Windows Security Event ID 4624 (Logon Type 10)

**KQL Query:**
```kql
event.code: 4624 AND winlog.event_data.LogonType: 10 AND NOT source.ip: "192.168.100.10"
```

**Expected Findings:** RDP sessions originating from non-IT subnets or from compromised workstations.

**Investigation Steps:**
1. Identify the source host. Is it a workstation, or is it the Kali box?
2. Check if the logged-in user account has an approved change request for remote access.
3. Correlate with Sysmon EID 3 on the source host to see what triggered the RDP session.

## Hunt 4: Living-off-the-Land Binaries (LOLBins) Making Network Connections

**Hypothesis:** Attackers may abuse legitimate Windows utilities (`certutil.exe`, `bitsadmin.exe`, `mshta.exe`) to download malicious payloads from the internet.

**Data Source:** Sysmon Event ID 3 (Network Connection)

**KQL Query:**
```kql
event.code: 3 AND process.name: ("certutil.exe" OR "bitsadmin.exe" OR "mshta.exe" OR "regsvr32.exe" OR "rundll32.exe") AND NOT destination.ip: ("10.*" OR "192.168.*")
```

**Expected Findings:** Any of these binaries making external network connections is a strong indicator of compromise.

**Investigation Steps:**
1. Extract the destination IP and URL from the Sysmon event.
2. Query the URL against VirusTotal.
3. Correlate with Sysmon EID 11 (File Create) to see if a file was downloaded.

## Hunt 5: Credential Dumping Indicators

**Hypothesis:** An adversary may have used Mimikatz or a similar tool to dump credentials from the LSASS process.

**Data Source:** Sysmon Event ID 10 (Process Access)

**KQL Query:**
```kql
event.code: 10 AND winlog.event_data.TargetImage: "*lsass.exe" AND NOT winlog.event_data.SourceImage: ("*\\MsMpEng.exe" OR "*\\csrss.exe" OR "*\\svchost.exe")
```

**Expected Findings:** Processes other than known system utilities accessing `lsass.exe`.

**Investigation Steps:**
1. Identify the `SourceImage` (the process that accessed LSASS).
2. Check its file hash against VirusTotal.
3. If confirmed malicious, the host's credentials must be considered fully compromised. Trigger a domain-wide password reset.

## Hunt 6: Suspicious Cron Job Creation (Linux Persistence)

**Hypothesis:** An attacker who has gained access to a Linux server may install a cron job to maintain persistence or exfiltrate data on a schedule.

**Data Source:** Filebeat (`/var/log/syslog`), Auditbeat (file integrity on `/etc/crontab`, `/var/spool/cron/`)

**KQL Query:**
```kql
message: "crontab" AND (message: "-e" OR message: "REPLACE")
```

**Expected Findings:** Newly created or modified cron jobs, especially ones running `wget`, `curl`, or reverse shell commands.

## Hunt 7: Unusual Service Installations

**Hypothesis:** Attackers frequently install malicious services (e.g., PsExec installs `PSEXESVC`) to execute payloads as SYSTEM.

**Data Source:** Windows System Event ID 7045

**KQL Query:**
```kql
event.code: 7045 AND NOT winlog.event_data.ServiceName: ("WinDefend" OR "wuauserv" OR "Spooler" OR "EventLog")
```

**Expected Findings:** Non-standard services installed recently.

**Investigation Steps:**
1. Review the service binary path (`ImagePath`). If it points to a temporary folder or has a randomly generated name, escalate.
2. Check the account the service runs as. If `LocalSystem`, the risk is critical.

## Hunt 8: Data Staging (Large File Compression)

**Hypothesis:** Before exfiltrating data, attackers often stage it by compressing large directories into ZIP or RAR archives.

**Data Source:** Sysmon Event ID 1 (Process Creation)

**KQL Query:**
```kql
event.code: 1 AND process.name: ("7z.exe" OR "rar.exe" OR "tar.exe" OR "Compress-Archive") AND process.command_line: (*Documents* OR *Desktop* OR *Finance* OR *HR*)
```

**Expected Findings:** Compression utilities targeting sensitive business directories.

## Hunt 9: Sudo Abuse on Linux

**Hypothesis:** A compromised low-privilege account may be attempting to escalate to root using `sudo`.

**Data Source:** Filebeat (`/var/log/auth.log`)

**KQL Query:**
```kql
message: "sudo" AND message: "NOT in sudoers"
```

**Expected Findings:** Accounts repeatedly denied by sudo, indicating an attacker probing for escalation paths.

**Investigation Steps:**
1. Identify the user account.
2. Check if it is a legitimate user who simply forgot their permissions, or a compromised service account.
3. Review all commands attempted by this user in the last 24 hours.

## Hunt 10: DNS Tunneling / Exfiltration

**Hypothesis:** Sophisticated attackers may use DNS queries to exfiltrate data or communicate with C2 infrastructure by encoding data within DNS subdomain requests.

**Data Source:** Sysmon Event ID 22 (DNS Query), Network Firewall Logs

**KQL Query:**
```kql
event.code: 22 AND dns.question.name: /.*\..{30,}\..*/
```

**Expected Findings:** DNS queries with unusually long subdomain strings (>30 characters), which may contain encoded data.

**Investigation Steps:**
1. Examine the full domain name. If it appears to be random alphanumeric characters, it is likely encoded data.
2. Identify the process making the DNS request.
3. Block the domain at the DNS firewall and isolate the host.
