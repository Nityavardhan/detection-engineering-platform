# SOC Dashboard Design Guide

**Version:** 1.0
**Date:** June 2026
**Role:** SOC Analyst / Detection Engineer

---

## 1. Overview
A SIEM is only as good as the SOC's ability to interpret its data. Dashboards must be designed with specific operational intents in mind. Throwing random charts on a screen leads to "Dashboard Fatigue."

This document outlines the three core dashboards required for our Enterprise SOC, their purpose, and the visualizations they contain.

---

## 2. Dashboard 1: Authentication & Access Monitoring
**Purpose:** To monitor for brute-force attacks, password spraying, and lateral movement across the enterprise.

### 2.1 Key Visualizations
- **Metric:** Total Failed Logins (Last 24h)
- **Metric:** Total Successful Logins (Last 24h)
- **Line Chart (Time Series):** Failed vs. Successful Logins over time. 
  - *Why:* A sudden spike in failed logins indicates an active brute-force or spraying attack.
- **Data Table:** Top Users with Failed Logins
  - *Why:* Identifies exactly which accounts are being targeted.
- **Data Table:** Top Source IPs for Failed Logins
  - *Why:* Identifies the attacker's infrastructure, which can be blocked at the firewall.
- **Pie Chart:** Logon Types (e.g., Network, Interactive, RDP)
  - *Why:* A spike in Logon Type 10 (Remote Interactive) might indicate unauthorized RDP access.

---

## 3. Dashboard 2: Process Execution & Threat Hunting
**Purpose:** To detect Living-off-the-Land (LOLBins) techniques, unauthorized administrative tools, and malware execution. Heavily relies on Sysmon EID 1 and Linux Auditbeat `execve`.

### 3.1 Key Visualizations
- **Data Table:** Most Active Processes (by count)
  - *Why:* Helps establish a baseline of normal activity.
- **Bar Chart:** PowerShell Executions by User
  - *Why:* PowerShell is an administrative tool; if a standard user (e.g., HR department) is executing it, that is highly suspicious.
- **Data Table:** Encoded Command Executions
  - *Why:* Attackers often use `powershell.exe -enc <base64>` to hide their payloads. We explicitly filter for the `-enc` or `-EncodedCommand` flags.
- **Metric:** Cleared Event Logs
  - *Why:* Tracks Windows Event ID 1102. Should always be zero. If > 0, immediate incident response is required.

---

## 4. Dashboard 3: Executive Summary (CISO View)
**Purpose:** To provide a high-level overview of SOC performance, risk, and alert volume to non-technical stakeholders.

### 4.1 Key Visualizations
- **Metric:** Total High/Critical Severity Alerts (Last 7 Days)
- **Bar Chart:** Top MITRE ATT&CK Tactics Detected
  - *Why:* Shows executives where the SOC is seeing the most activity (e.g., Initial Access vs. Lateral Movement).
- **Pie Chart:** Alerts by Compliance Framework (ISO 27001, NIST)
  - *Why:* Demonstrates ROI (Return on Investment) to leadership by proving that the SOC is actively monitoring required compliance controls.
- **Data Table:** Mean Time To Detect (MTTD) & Mean Time To Respond (MTTR)
  - *Why:* Core SOC KPIs.

---

## 5. Implementation Notes
When building these in Kibana:
1. Always use **Lens** for building modern visualizations.
2. Ensure the time picker is defaulted to **Last 24 Hours** for operational dashboards, and **Last 7 Days** for Executive dashboards.
3. Apply filters aggressively. Do not query `*`; query specific Event IDs to save compute resources on the Elasticsearch cluster.
