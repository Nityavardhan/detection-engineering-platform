# Incident Response Playbook Library (Part 1 of 2)

**Version:** 1.0
**Date:** June 2026
**Framework:** NIST SP 800-61 Rev. 2 (Computer Security Incident Handling Guide)

---

## Overview

Every playbook in this library follows the **NIST Incident Response Lifecycle**:

1. **Preparation** — What tools, access, and contacts must be ready before the incident occurs.
2. **Identification** — How to confirm the alert is a true positive.
3. **Containment** — How to stop the bleeding without destroying evidence.
4. **Eradication** — How to remove the root cause from the environment.
5. **Recovery** — How to restore normal operations.
6. **Lessons Learned** — What post-incident review to conduct.

---

## Playbook 1: Brute Force / Password Spraying Incident

**Trigger:** High volume of Event ID 4625 (Failed Logon) from a single source IP within 10 minutes.

### Preparation
- Ensure account lockout policies are configured (e.g., lockout after 5 failed attempts).
- Pre-configure a "Block IP" action in the firewall management console.

### Identification
1. Query Kibana: `event.code: 4625 | stats count() by source.ip | where count > 50`.
2. Confirm the source IP is not an internal service account or load balancer.
3. Check if any Event ID 4624 (Successful Logon) followed the failures from the same IP—if so, escalate to **Critical**.

### Containment
- **Short-term:** Block the attacking IP at the perimeter firewall immediately.
- **Long-term:** If a successful login occurred, disable the compromised account and force a password reset.

### Eradication
- Review all active sessions for the compromised account and terminate them.
- Scan the host the account logged into for malware or persistence mechanisms.

### Recovery
- Re-enable the account with a new, strong password.
- Enable Multi-Factor Authentication (MFA) on the account.

### Lessons Learned
- Was the account targeted because it had a weak password? Enforce password complexity policies.
- Was MFA already enabled? If not, why not?

## Playbook 2: Compromised User Account

**Trigger:** Anomalous login pattern (e.g., login from an unusual country, at 3:00 AM, via VPN + simultaneous internal login).

### Identification
1. Correlate Event ID 4624 with GeoIP data. Is the source IP from a country the user has never logged in from?
2. Check for "impossible travel"—two logins from geographically distant locations within a short timeframe.
3. Review Sysmon EID 1 for any suspicious process execution under the user's context.

### Containment
- Disable the user account immediately.
- Revoke all active OAuth tokens and SSO sessions.
- Notify the user through an out-of-band channel (e.g., phone call) to confirm they are not the source.

### Eradication
- Force a password reset.
- Review all files accessed and emails sent from the account during the compromised window.

### Recovery
- Re-enable the account after the user confirms the new password via their manager.
- Enable MFA.

### Lessons Learned
- How did the attacker obtain the credentials? Phishing? Credential stuffing from a data breach?

## Playbook 3: Malware Execution on Endpoint

**Trigger:** Sysmon EID 1 showing execution of an unknown binary, or Windows Defender alert (Event ID 1116/1117).

### Identification
1. Collect the file hash from Sysmon EID 1 (`process.hash.sha256`).
2. Query the hash against VirusTotal and internal Threat Intelligence.
3. Check Sysmon EID 3: Did the malware make outbound network connections (C2)?
4. Check Sysmon EID 11: Did the malware drop additional files?

### Containment
- **Isolate the host from the network** (disable NIC or use EDR network isolation).
- Do NOT power off the machine—volatile memory evidence will be lost.

### Eradication
- Use EDR or Defender to quarantine the malware binary.
- Remove any persistence mechanisms (registry keys, scheduled tasks, startup folder items).
- Scan the entire disk with an offline antivirus scanner.

### Recovery
- Re-image the machine if confidence in eradication is low.
- Restore user data from known-good backups.
- Reconnect to the network and monitor closely for 72 hours.

### Lessons Learned
- How did the malware arrive? Email attachment? USB drive? Drive-by download?
- Was the user running as a local administrator? If so, enforce least-privilege.

## Playbook 4: Ransomware Incident

**Trigger:** Multiple Sysmon EID 11 (File Create) events showing mass file renaming with unusual extensions (e.g., `.encrypted`, `.locked`).

### Identification
1. Confirm the file renaming pattern across multiple directories.
2. Check for a ransom note file (e.g., `README_DECRYPT.txt`).
3. Identify Patient Zero—the first host that exhibited the behavior.

### Containment
- **Immediately disconnect the affected host(s) from the network.**
- **Disable shared drives (SMB/CIFS)** to prevent lateral encryption.
- Alert the executive team and legal counsel.

### Eradication
- Identify the ransomware variant using the ransom note or encrypted file samples (e.g., via ID Ransomware).
- Determine if a free decryptor exists.
- Remove the ransomware binary and all persistence mechanisms.

### Recovery
- Restore from the most recent clean backup. **Do not pay the ransom.**
- Re-image all affected hosts.
- Reset all domain credentials.

### Lessons Learned
- Was the initial access vector a phishing email? Strengthen email filtering.
- Were backups air-gapped? If they were encrypted too, implement offline backups.

## Playbook 5: Privilege Escalation Detected

**Trigger:** Event ID 4732 (User added to Administrators group) or Sysmon EID 1 showing execution of known escalation tools.

### Identification
1. Determine which account was added to the privileged group and by whom.
2. Cross-reference with the change management system—was this an approved change?
3. If the adding account is also newly created (EID 4720), treat as a compromise chain.

### Containment
- Remove the account from the privileged group immediately.
- Disable both the escalated account and the account that performed the escalation.

### Eradication
- Audit all actions performed by the escalated account during the privilege window.
- Check for additional persistence (scheduled tasks, services, registry keys).

### Recovery
- Re-enable accounts only after a full audit and password reset.
- Implement Just-In-Time (JIT) privileged access if available.

## Playbook 6: Event Log Clearing

**Trigger:** Event ID 1102 (Security Log Cleared) or Event ID 104 (System Log Cleared).

### Identification
1. This alert has **zero acceptable false positives** in a governed environment. Treat as a true positive immediately.
2. Identify the account and host that cleared the logs.
3. Since the local logs are now gone, rely on the SIEM's copy (Elasticsearch) to reconstruct the timeline *before* the clearing.

### Containment
- Isolate the host from the network.
- Disable the account that cleared the logs.

### Eradication
- Reconstruct the full attack timeline from Elasticsearch indices (which still have all the forwarded logs).
- Search for the root cause—what did the attacker do that they needed to hide?

### Recovery
- Re-image the host.
- Implement a GPO that prevents non-SYSTEM accounts from clearing Security logs.

## Playbook 7: Lateral Movement (PsExec / RDP)

**Trigger:** Event ID 7045 (PSEXESVC service installed) or Event ID 4624 Logon Type 10 from a non-IT subnet.

### Identification
1. Identify the source and destination hosts.
2. Determine the credentials used—is it a domain admin account?
3. Check the source host for prior compromise indicators (malware, credential dumping).

### Containment
- Isolate both the source and destination hosts.
- Disable the credentials used for lateral movement.
- Block the source IP from reaching other sensitive segments.

### Eradication
- Remove the PSEXESVC service from the destination host.
- Scan both hosts for malware and persistence.
- Reset the compromised credentials.

### Recovery
- Reconnect hosts after clean scans.
- Implement network segmentation to prevent future unrestricted lateral movement.

## Playbook 8: Web Shell Detected (Linux)

**Trigger:** Auditbeat detecting `bash` or `sh` spawned as a child of `nginx`, `apache2`, or `php-fpm`.

### Identification
1. Confirm the web server process spawned an interactive shell—this should never happen in production.
2. Check web server access logs for file upload activity immediately preceding the shell execution.
3. Identify the uploaded web shell file (often a `.php` or `.jsp` file in the web root).

### Containment
- Firewall the server to block all inbound traffic except from the SOC's investigation IP.
- Do NOT restart the web server—this may trigger the attacker's dead man's switch.

### Eradication
- Remove the web shell file from the web root.
- Patch the web application vulnerability that allowed the upload.
- Review all commands executed through the shell (Auditbeat `execve` logs).

### Recovery
- Deploy a patched version of the web application.
- Restore the web root from a known-good backup.
- Implement a Web Application Firewall (WAF) to block file upload exploits.
