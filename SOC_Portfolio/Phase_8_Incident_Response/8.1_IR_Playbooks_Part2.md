## Playbook 9: Credential Dumping (Mimikatz / LSASS Access)

**Trigger:** Sysmon EID 10 showing an unknown process accessing `lsass.exe` with `GrantedAccess: 0x1010` or `0x143a`.

### Identification
1. Identify the `SourceImage`—the binary that accessed LSASS.
2. Hash the binary and query VirusTotal. Known hashes for Mimikatz will return immediate positives.
3. Assume **all credentials on the compromised host are stolen**.

### Containment
- Isolate the host immediately.
- Force a password reset for **every account that has ever logged into that machine** (cached credentials are now compromised).
- If domain admin credentials were cached, initiate a **domain-wide password reset** and reset the KRBTGT account twice.

### Eradication
- Remove the credential dumping tool.
- Check for Golden Ticket / Silver Ticket persistence (Kerberos artifacts).
- Audit all systems the compromised credentials had access to.

### Recovery
- Re-image the compromised host.
- Implement Credential Guard and LSA Protection on all endpoints.

### Lessons Learned
- Why did the attacker have local admin privileges? Enforce least-privilege.
- Was Credential Guard enabled? If not, add it to the hardening baseline.

## Playbook 10: Phishing Email Reported

**Trigger:** User reports a suspicious email to the SOC, or email gateway flags a malicious attachment/URL.

### Identification
1. Analyze the email headers to identify the true sender (check `Reply-To`, `Return-Path`, SPF/DKIM results).
2. If there is an attachment, extract the hash and query VirusTotal. **Do NOT open it on your workstation.**
3. If there is a URL, query it against URLhaus and VirusTotal.
4. Search the email gateway for all recipients who received the same email.

### Containment
- Purge the email from all inboxes across the organization.
- Block the sender domain and any malicious URLs at the email gateway and DNS firewall.
- If any user clicked the link or opened the attachment, treat their host as compromised (see Playbook 3: Malware Execution).

### Eradication
- Reset credentials for any user who interacted with the phishing email.
- Scan affected hosts for malware.

### Recovery
- Restore any quarantined legitimate emails that were caught by the tightened filters.
- Send an organization-wide awareness bulletin about the phishing campaign.

## Playbook 11: Data Exfiltration Suspected

**Trigger:** Sysmon EID 1 showing compression utilities (7z, rar) targeting sensitive directories, or unusually large outbound data transfers detected by network monitoring.

### Identification
1. Identify the user and host performing the compression/transfer.
2. Determine what data was compressed (check `process.command_line` for directory paths).
3. Correlate with Sysmon EID 3 for outbound connections immediately after the compression.

### Containment
- Block the destination IP/domain at the firewall.
- Disable the user account.
- Preserve the compressed archive as forensic evidence before deleting.

### Eradication
- Determine the initial access vector—how did the attacker get into the network?
- Remove all persistence mechanisms.

### Recovery
- Assess the business impact of the exfiltrated data.
- Notify legal counsel and, if PII/PHI was involved, initiate breach notification procedures.

## Playbook 12: Insider Threat

**Trigger:** Anomalous data access patterns by an employee (e.g., accessing files outside their department, bulk downloads from SharePoint).

### Identification
1. Correlate file access logs with the user's role. Is a Finance employee accessing Engineering source code?
2. Check for USB device insertions (Sysmon EID 6 - Driver Loaded, or USB audit logs).
3. Review the user's recent email for large attachments sent to personal email addresses.

### Containment
- Do NOT alert the user. Coordinate with HR and Legal first.
- Increase monitoring on the user's account (enhanced logging, screen recording if legally permitted).

### Eradication
- If confirmed malicious, disable the account and revoke all access.
- Preserve all evidence for potential legal proceedings.

### Recovery
- Assess what data was accessed or exfiltrated.
- Conduct a company-wide access review to ensure role-based access control (RBAC) is properly configured.

## Playbook 13: Denial of Service (DoS/DDoS)

**Trigger:** Network monitoring detects a massive spike in inbound traffic, or web applications become unresponsive.

### Identification
1. Determine if the attack is volumetric (bandwidth exhaustion) or application-layer (HTTP flood).
2. Identify the source IPs (often spoofed in DDoS attacks).
3. Check with the ISP for upstream traffic patterns.

### Containment
- Enable rate-limiting at the load balancer / WAF.
- Activate DDoS mitigation services (e.g., Cloudflare, AWS Shield) if available.
- Null-route the most aggressive source IPs.

### Eradication
- DDoS attacks typically stop on their own once the attacker's resources are exhausted.
- Investigate if the DDoS was a smokescreen for a simultaneous intrusion attempt.

### Recovery
- Restore normal traffic flow once attack subsides.
- Conduct a capacity review to ensure infrastructure can handle future surges.

## Playbook 14: Unauthorized Firewall Changes

**Trigger:** Sysmon EID 1 detecting `netsh.exe advfirewall set allprofiles state off`, or firewall management logs showing policy changes.

### Identification
1. Identify the user and host that made the change.
2. Cross-reference with the change management system—was this approved?
3. Determine if the firewall was disabled to facilitate data exfiltration or lateral movement.

### Containment
- Re-enable the firewall immediately.
- Isolate the host if the change was unauthorized.

### Eradication
- Determine why the firewall was disabled. Was it part of a larger attack chain?
- Check for malware or C2 activity that occurred while the firewall was down.

### Recovery
- Implement GPO-enforced firewall policies that prevent local administrators from disabling the firewall.
- Audit all traffic that left the host during the window the firewall was off.

## Playbook 15: Security Tool Tampering (Defender Disabled)

**Trigger:** Sysmon EID 1 showing `Set-MpPreference -DisableRealtimeMonitoring $true` or Defender Event ID 5001 (Real-time protection disabled).

### Identification
1. Determine who or what disabled Defender. Was it a user, a script, or malware?
2. Check for subsequent malware execution (Sysmon EID 1 for suspicious binaries launched *after* Defender was disabled).
3. This is almost always a precursor to malware deployment—treat as high priority.

### Containment
- Re-enable Windows Defender Real-Time Protection immediately.
- Isolate the host from the network.

### Eradication
- Scan the host with an alternative antivirus engine (since Defender was tampered with).
- Remove any malware and persistence mechanisms.

### Recovery
- Implement Tamper Protection (Windows Security setting) to prevent Defender from being disabled even by local administrators.
- Add a GPO that enforces Defender settings centrally.

### Lessons Learned
- Was Tamper Protection already enabled? If the attacker bypassed it, escalate to Microsoft support.
- Ensure all endpoints have Tamper Protection enforced via Intune or GPO.
